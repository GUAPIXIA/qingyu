三、架构评估

3.1 整体架构

项目采用 Electron + React 标准三层架构：

  • 渲染进程（src/）：React 18 + Zustand 5 + react-router-dom 7（HashRouter），UI 用 Tailwind CSS 3，虚拟滚动用 react-virtuoso
  • 主进程（electron/）：14 个 IPC 模块 + 服务层（AI 适配器、存储、生图、TTS、MCP）
  • 服务端（server/）：可选独立部署的公告系统，Express + better-sqlite3 + JWT
  • 共享层（shared/）：types.ts 类型定义 + ipc-api.ts IPC 契约

整体分层清晰，技术栈选型合理。但存在以下架构问题：

移除所有硬编码凭据，要求环境变量注入：JWT_SECRET（≥32 字节）、ADMIN_PASSWORD、ALLOWED_ORIGINS。docker-compose.yml 使用 env_file。提供 .env.example 模板。首次登录强制改密。

6.2 用户体验优化

危险操作确认弹窗统一 重要

当前删除消息、删除 MCP Server、清空群聊、删除 API 配置等都直接执行无确认。统一使用已有的 ConfirmDialog 组件。

全局错误通知系统 重要

当前多处 catch {} 完全吞掉错误（重新生图、handleBranch、GroupChatPage 导出等）。建立统一的 toast/notification 机制，所有错误对用户可见。

加载状态完善 建议

PersonasPage、LorebookPage、PresetsPage 首次加载显示空状态而非 loading 指示。AI 续写/润色/翻译过程应有进度反馈。重新生成按钮应有 loading 视觉反馈。

可访问性 (a11y) 建议

Modal 添加 role="dialog" + aria-modal + 焦点陷阱 + 焦点恢复；Toggle 添加 role="switch" + aria-checked；所有图标按钮添加 aria-label；命令补全下拉添加 listbox 语义；Esc 关闭图片预览和上下文查看器。

角色列表虚拟化 建议

CharactersPage 和 LorebookPage 条目列表在数据量大（100+）时未虚拟化会卡顿。复用已有的 react-virtuoso。

6.3 功能扩展

多窗口/多会话并行 新功能

支持多个独立聊天窗口同时打开，方便用户同时与多个角色对话。Electron 的 BrowserWindow 可轻松实现，需要重构 store 支持多实例。

角色卡市场 / 在线分享 新功能

复用已有的公告服务端基础设施，扩展为角色卡分享平台。用户可上传/下载/评分角色卡，配套审核机制。

对话搜索与全文索引 新功能

当前无法搜索历史消息。引入本地全文索引（如 SQLite FTS5 或 lunr.js），支持跨会话、跨角色搜索关键词，高亮匹配。

消息编辑历史 / 版本对比 新功能

当前消息编辑直接覆盖。引入版本历史（类似 git），可查看/回滚任意历史版本，对比 diff。

统计与可视化增强 新功能

扩展 UsagePage：按角色/会话维度统计、费用趋势图、Token 消耗热力图、消息长度分布。引入 Chart.js 或 Recharts。

插件系统 新功能

基于现有 MCP 架构扩展，允许用户安装第三方插件扩展斜杠命令、消息处理器、UI 面板。需要沙箱隔离机制。

跨设备同步（可选） 新功能

当前完全本地存储。提供可选的端到端加密同步，用户可在多设备间同步角色/对话/设置（需解决 API Key 加密同步问题）。

6.4 技术债务清理

重构 useGroupChatStore 重要

该文件集中了 7 处高严重度 Bug（已修复 5 处），是技术债最重的文件。建议以 useChatStore 的 streamAIResponse + cleanupActiveStream + flushStream 为模板，抽取公共 createStreamController 工厂，群聊和单聊复用。

拆分大型组件 重要

GroupChatPage（1114 行）、ChatPage（899 行）、ai.ts（888 行）、CharacterEditor（739 行）、ChatInput（739 行）等都应拆分。每个文件不超过 300-400 行为佳。

存储层迁移到 SQLite 建议

当前消息存储用 JSONL 文件，存在：非原子写入、全量读取性能差、无索引、无事务。迁移到 better-sqlite3（已在服务端使用）可解决：原子事务、索引查询、FTS 全文搜索、并发安全。

类型系统完善 建议

移除 eslint-disable no-explicit-any，为 AI 响应定义 interface；shared/ipc-api.ts 的 McpAPI/UsageAPI/LogAPI 用具体类型替换 any；useGroupChatStore 的辅助函数用 StoreSet/StoreGet 类型。

添加测试 建议

当前项目无任何测试。建议：核心工具函数单元测试（tokenCounter、dialogue-parser、variables）；store action 集成测试（mock IPC）；关键 IPC handler 测试。引入 vitest + @testing-library/react。

统一错误边界 建议

整个应用没有 ErrorBoundary，任何渲染期错误都会白屏。在 main.tsx 包裹全局 ErrorBoundary，关键页面单独包裹局部 ErrorBoundary。
